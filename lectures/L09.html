s9.1 — Landmark Detection & Pose Estimation

What is landmark detection?

Why it matters: landmark detection is the foundation for facial-recognition, emotion detection, AR filters (Snapchat-style hats / glasses), and human-pose estimation. It is also the gentlest entry point into "predict coordinates with a CNN" — every later detector (R-CNN, YOLO) reuses the same idea of regressing $(x,y)$ pairs.

Landmark detection = given an image of an object (face, body, hand…), output a fixed-length vector of $(x,y)$ coordinates that mark anatomically meaningful points on that object.

It is a regression problem (continuous outputs) — not a classification problem.

The output vector $Y$ — coordinates only

Why it matters: this exact vector shape comes back in every keypoint network. Memorise the layout.

For an object with $n$ landmarks, the network outputs a vector with $2n$ numbers — the $x$ and $y$ of each landmark, stacked:

$$Y \;=\; [\,L_{1x},\; L_{1y},\; L_{2x},\; L_{2y},\; \dots,\; L_{nx},\; L_{ny}\,]$$

Symbols: $L_{ix}$ = x-coordinate of landmark $i$, $L_{iy}$ = y-coordinate of landmark $i$, $n$ = number of landmarks (fixed per task: e.g. 32 for the slide example, 68 for dlib face, 17 for COCO pose).

The output vector $Y$ — with the "object present" flag $p_c$

Why it matters: the network must also tell us "is there even a face/person here?" before its coordinate predictions can be trusted. So we prepend a binary flag $p_c$ (probability the class is present).

$$Y \;=\; [\,p_c,\; L_{1x},\; L_{1y},\; L_{2x},\; L_{2y},\; \dots,\; L_{nx},\; L_{ny}\,]$$

Length is $1 + 2n$. If $p_c \approx 0$, you ignore the coordinate part (it's garbage); if $p_c \approx 1$, the coordinates are meaningful.

Worked example — 68-landmark face detector A 68-point face model outputs $p_c$ + 68 $\times$ $(x,y)$ pairs:
length = $1 + 2 \cdot 68 = 1 + 136 = \mathbf{137}$.
For a 32-landmark face: $1 + 64 = 65$.
For a 17-keypoint pose: $1 + 34 = 35$.

Common exam traps

  • "68 landmarks → output length 68." — False. Each landmark is 2 numbers, so 136, plus $p_c$ = 137.
  • "$p_c$ is the class label." — No. It is a presence/objectness probability for a single class; multiclass landmarks need separate detectors per class.
  • "Landmark detection is classification." — No, it is regression on the coordinates (with one optional classification head for $p_c$).

Pose estimation — same idea, body keypoints

Why it matters: human-pose estimation is just landmark detection where the object is a person and the landmarks are joints. Same vector layout.

OpenPose library & datasets

Why it matters: the exam may ask which library / dataset is canonical for human-pose estimation.

OpenPose — real-time, multi-person 2D keypoints (15 / 18 / 27 body+foot variants, 21 hand keypoints, 70 face keypoints), 3D from multi-view triangulation, single-person tracking, camera calibration toolbox. License free for non-commercial use; commercial fee USD 25 000/year.

Pose datasets to know: MPII Human Pose, 3DPW, LSPe (Leeds Sports Pose Extended), DensePose-COCO, AMASS, VGG Human Pose, SURREAL, CMU Panoptic, COCO (the canonical one), HumanEva, Human3.6M, FLIC, 300-W, Labeled Face Parts in the Wild.

Q. A network is trained on 21 hand keypoints with a presence flag. What is the output dimensionality?
Answer$1 + 2\cdot 21 = 43$.
Q. The 70-face-keypoint OpenPose model outputs how many numbers per face (no $p_c$ assumed)?
Answer$2\cdot 70 = 140$.

s9.2 — R-CNN (Region-based CNN)

Quick review — sliding window detection

Why it matters: R-CNN exists to fix the sliding-window bottleneck. You have to understand the problem to see why R-CNN is better.

Classical detection: slide a window of size 1 across the image, then a window of size 2, then size 3… extract the sub-image under each window, feed it to a classifier (SVM, CNN), and remember the windows that fire.

Main disadvantage: very slow. Two reasons (slide 21):

  1. The number of picked windows is huge (multi-scale × multi-position ⇒ thousands to tens-of-thousands per image).
  2. For a CNN classifier you must run a full convolution on every window's content.

R-CNN core idea — propose, then classify

Why it matters: R-CNN was the 2014 paper that brought CNNs to object detection. Replacing brute-force sliding-window with a smart region proposer reduced the candidate count from ~tens of thousands to ~2000.

Main idea: instead of feeding every possible sliding window to a detector, first propose a small set of candidate regions that are likely to contain an object — then run the detector only on those.

Pipeline (slide 23):

  1. Input image.
  2. Region-proposal algorithm produces up to ~2000 candidate boxes (each described by $x, y, W, H$).
  3. Each candidate is fed to an Object Detector.
  4. Detector outputs: refined bounding box (a regressor adjusts $x,y,W,H$) and object class.

Region proposal — Selective Search & Edge Boxes

Why it matters: the proposal step is where R-CNN diverges from sliding window. The exam loves "name two region-proposal techniques".

The two most common hand-crafted region-proposal techniques are:

  1. Selective Search — hierarchical merging of small image segments based on similarity of colour, intensity, texture, size and fill. Start from super-pixels (small uniform regions) and greedily merge the most similar neighbouring regions, recording each merged region as a proposal at every scale.
  2. Edge Boxes — uses edge maps; a box scores high when many edges are fully enclosed by it (closed contours = candidate objects), and low when edges leave the box. Fast and complementary to selective search.

In R-CNN, each proposed region is then cropped from the input image and warped to a fixed size (e.g. 224×224) before being passed through the CNN — every region must produce a same-size feature map so the FC head accepts it.

Selective Search — the four similarity measures (mnemonic)
  1. Colour similarity (histogram intersection in RGB or HSV).
  2. Texture similarity (gradient histograms — like SIFT/HOG-style).
  3. Size similarity (prefer merging small regions first → balanced hierarchy).
  4. Fill similarity (regions that, when merged, leave little empty space inside their bounding box).
Combine $s = a_1 s_\text{colour} + a_2 s_\text{texture} + a_3 s_\text{size} + a_4 s_\text{fill}$.

The full R-CNN pipeline (the iconic 4-step diagram)

Why it matters: this is the diagram you should be able to draw on the exam from memory.

  1. Input image.
  2. Extract ~2000 region proposals with Selective Search (or Edge Boxes).
  3. Warp each region to a fixed input size and compute CNN features by running the full CNN on it. This step is the bottleneck — you run the CNN ~2000 times per image.
  4. Classify regions — one class-specific linear SVM per class scores the CNN feature (aeroplane? no … person? yes … tv-monitor? no …). A separate bounding-box regressor refines the box.
What gets trained in R-CNN (multi-stage!)
  1. Pre-train the CNN backbone on ImageNet classification.
  2. Fine-tune the CNN on warped region crops (object vs background).
  3. Train one linear SVM per class on the CNN features (object class vs not-that-class).
  4. Train a per-class bounding-box regressor (linear regression on $x,y,W,H$ refinements).
Four separate training stages → ugly to engineer, easy to mis-tune. Fast R-CNN fixes this.

Disadvantages of R-CNN — and the path forward

Why it matters: each disadvantage points directly at a fix in Fast / Faster R-CNN.

  1. Cropping the ~2000 proposed regions is itself slow.
  2. Multiple convolutions — convolution must be applied to every proposed image (~2000 forward passes per training/inference image).
  3. The same object will be detected in multiple overlapping windows because proposals are not optimised; we need Non-Max Suppression to clean up.

Solution to reason #2 (the biggest cost): save time by running the CNN one time only on the whole input image, then reuse that single feature map for every proposed region. That insight is exactly Fast R-CNN.

Common exam traps

  • "R-CNN uses softmax classification." — False. R-CNN uses a separate SVM per class on top of CNN features. (Fast R-CNN is the one that uses softmax.)
  • "R-CNN has end-to-end training." — False. R-CNN training is multi-stage (CNN fine-tune, SVMs, bbox regressors trained separately).
  • "R-CNN proposes ~2k regions per object." — No, ~2000 per image total.
  • "Selective Search runs on a GPU." — No. Classical Selective Search runs on CPU; that's why it remains a bottleneck even after Fast R-CNN.
Q. Why does R-CNN need to warp every proposal to a fixed size before the CNN?
AnswerBecause the FC head of the CNN expects a fixed-length feature vector. Different region sizes would produce different feature-map sizes, which the FC layer cannot accept.
Q. R-CNN runs the CNN forward pass roughly how many times per image at inference?
Answer~2000 — once per proposed region.

s9.3 — Fast R-CNN (Extract Regions from Feature-Map Domain)

Core trick — one CNN pass on the whole image

Why it matters: Fast R-CNN is roughly an order of magnitude faster than R-CNN at inference, with comparable accuracy. The trick — share computation across proposals — is the most important architectural idea in modern detection.

  • Region proposals still come from Selective Search (same as R-CNN — this is the bit that Faster R-CNN will replace later).
  • The whole image is passed through the CNN once to get a feature map of the whole image.
  • Each region is then extracted from the feature map, not from the original image — i.e. the proposal's bounding box is projected onto the feature map.
  • Different proposals → different-sized feature-map slices, which is incompatible with the fixed-size FC head. The fix is RoI pooling: a special pooling layer that takes any input size and produces a fixed-size output (e.g. 7×7).
  • The fixed-length RoI features feed an FC network that produces both classification (softmax over classes) and bounding-box regression.

The Fast R-CNN diagram — RoI projection & RoI pooling

RoI pooling — concretely, what does it do? Given a proposed region projected onto the feature map (say a $14\!\times\!21$ patch) and a target size (say $7\!\times\!7$), RoI pooling splits the patch into a $7\!\times\!7$ grid of roughly equal sub-windows and applies max-pool inside each sub-window. The result is a $7\!\times\!7$ feature for every proposal, regardless of input size. (Conceptually: "adaptive max pool to a fixed grid".)
Worked example — projecting an RoI onto the feature map Suppose the backbone has cumulative stride 16 (e.g. VGG-16 conv5). The image is 800×800, so the feature map is 50×50. A selective-search proposal at image coords $(x_1, y_1, x_2, y_2) = (160, 320, 480, 480)$ projects to feature-map coords $(\lfloor 160/16 \rfloor, \lfloor 320/16 \rfloor, \lfloor 480/16 \rfloor, \lfloor 480/16 \rfloor) = (10, 20, 30, 30)$ — i.e. a 20×10 patch on the feature map. RoI pool then squashes that 20×10 patch into 7×7 by max-pooling.

Multi-task loss — classification + smooth-L1 box

Why it matters: Fast R-CNN trains in a single stage with one loss. That single-stage training (vs R-CNN's 4 stages) is half the speed-up story.

For each RoI, the FC head produces (a) a softmax distribution $p$ over $K+1$ classes (the +1 is "background"), and (b) a bounding-box refinement $t = (t_x, t_y, t_w, t_h)$.

Multi-task loss:

$$L \;=\; L_\text{cls} \;+\; \lambda\, L_\text{box}$$

where

  • $L_\text{cls} = -\log p_{u}$ — log-loss for the true class $u$ (standard softmax cross-entropy).
  • $L_\text{box} = \sum_{i\in\{x,y,w,h\}} \text{smooth}_{L_1}(t_i - t_i^\ast)$ — sum of smooth-$L_1$ on the four box offsets, computed only when $u\ge 1$ (i.e. the RoI is not background; for background the box loss is gated to zero).
  • $\lambda$ balances the two terms (typically 1).

Smooth-L1 is the Huber-style robust loss:

$$\text{smooth}_{L_1}(x)=\begin{cases}0.5\,x^2 & |x|<1 \\ |x|-0.5 & \text{otherwise}\end{cases}$$

It is less sensitive to outliers than plain $L_2$ — large box mistakes don't dominate the gradient.

Common exam traps

  • "Fast R-CNN runs Selective Search on the GPU." — False. Selective Search is still a CPU algorithm; this is exactly why Fast R-CNN is still proposal-bound (next section).
  • "RoI pooling crops the input image." — False. It crops a region from the feature map, then pools to a fixed size.
  • "Fast R-CNN uses an SVM." — False. It uses softmax.
  • "Fast R-CNN trains in 4 stages." — False. One stage, one multi-task loss.
Q. Two Fast R-CNN proposals on the same image have different sizes (200×100 px vs 80×80 px). After RoI pooling (target 7×7), what are the two output feature shapes?
AnswerBoth are $7\!\times\!7\!\times C$ (same shape) — that's the whole point of RoI pooling.
Q. Fast R-CNN's loss is $L = L_\text{cls} + \lambda L_\text{box}$. For a background RoI ($u=0$), what is $L$?
AnswerOnly the classification term — the box loss is gated to zero when the class is background, because background has no meaningful box.

s9.4 — Why Faster R-CNN? (motivation for L10)

The remaining bottleneck — Selective Search

Why it matters: Fast R-CNN sped up the detector, but Selective Search itself still runs on the CPU and now dominates the inference time. If we can make the proposal step learnable and GPU-friendly, the whole pipeline becomes ≈ real-time.

Disadvantages of Fast R-CNN:

  1. Slow — because Selective Search runs on the CPU and produces ~2000 proposals per image; the proposal step is now the bottleneck (the CNN step is fast).
  2. Still a large number of proposed regions.
  3. The same object is still detected in multiple boxes — needs NMS post-processing.

Solution:

  • Automate region proposal using a trained network (instead of hand-crafted Selective Search / Edge Boxes). This is the Region Proposal Network (RPN) introduced in Faster R-CNN.
  • Optimise the number of proposed regions (only ~300 from the RPN vs ~2000 from Selective Search) — fewer regions, higher recall.

Faster R-CNN — preview

Why it matters: L10 is dedicated to Faster R-CNN; here we only motivate it.

Read the architecture diagram (slide 35) left → right:

  1. Input image → CONV backbone → feature maps (shared with both the RPN and the detector head — note the red ✗ on Selective Search, the lecturer's own annotation).
  2. The shared feature map feeds the RPN — for each spatial location it outputs (a) objectness classification (object vs background) and (b) a bounding-box regressor.
  3. RPN proposals (now learned, only ~300 of them) are projected back onto the feature map.
  4. RoI pooling as in Fast R-CNN.
  5. FC detector head: multiclass classification + bbox regressor per RoI.

Common exam traps

  • "Faster R-CNN replaces Fast R-CNN's RoI pooling." — False. Faster R-CNN keeps RoI pooling unchanged; it replaces only the proposal step (Selective Search → RPN).
  • "Faster R-CNN does not need a region proposal step." — False. It still has region proposals; they just come from a trained network.
  • "Faster R-CNN's RPN replaces the backbone CNN." — False. The RPN shares the backbone with the detector head.
  • "R-CNN and Fast R-CNN both use softmax." — False. R-CNN uses SVMs; Fast R-CNN was the first to switch to softmax.
Q. Why is Fast R-CNN still slow even though it runs the CNN only once per image?
AnswerBecause Selective Search (which generates the ~2000 proposals) is a CPU algorithm and now dominates the per-image latency — the proposal step is the new bottleneck.
Q. In one sentence, what is the single key idea of Faster R-CNN that makes it faster than Fast R-CNN?
AnswerReplace the hand-crafted Selective Search proposal step with a small CNN (the Region Proposal Network) that shares features with the detector backbone, so proposals are learned on the GPU.

Practice quiz — Lecture 9

1 (Numerical). A landmark-detection network is trained for a face with 68 landmarks and includes the object-presence flag $p_c$. What is the dimensionality of its output vector $Y$?
Answer$1 + 2\cdot 68 = \mathbf{137}$. Layout: $Y=[p_c, L_{1x}, L_{1y}, \dots, L_{68x}, L_{68y}]$.
2 (MCQ). Which of the following is not a similarity used by Selective Search?
a) colour   b) texture   c) size   d) optical flow
Answerd) optical flow. Selective Search uses colour, texture, size and fill.
3 (MCQ — the big comparison). Which row is wrong?
a) R-CNN runs the CNN once per proposal.
b) Fast R-CNN crops regions from the feature map, not the image.
c) Fast R-CNN uses an SVM per class.
d) Faster R-CNN uses an RPN instead of Selective Search.
Answerc) is wrong. Fast R-CNN uses a softmax head; the SVM-per-class is R-CNN.
4 (T/F). Faster R-CNN no longer needs a region proposal step.
AnswerFalse. It still proposes regions — but with a trained Region Proposal Network (RPN) sharing features with the detector backbone.
5 (Short). Write the Fast R-CNN multi-task loss and name each term.
Answer$L = L_\text{cls} + \lambda L_\text{box}$, where $L_\text{cls}$ is softmax cross-entropy over $K+1$ classes (the +1 is background) and $L_\text{box}$ is the sum of smooth-$L_1$ losses on $(t_x,t_y,t_w,t_h)$, active only when the RoI is not background.
6 (Short). In one sentence, why is Fast R-CNN faster than R-CNN at inference?
AnswerBecause the heavy convolutional backbone runs once per image on the whole image (proposals are projected onto the shared feature map) instead of once per ~2000 proposals.
7 (Scenario). You are building a real-time pedestrian detector on a phone. R-CNN gives you ~47 s/image; Fast R-CNN ~2 s; Faster R-CNN ~0.2 s. Which step makes Faster R-CNN faster than Fast R-CNN, and why does it work?
AnswerThe Region Proposal Network. It is a small fully-convolutional network that shares the backbone feature map with the detector — so generating ~300 high-quality proposals costs only a few milliseconds on the GPU, whereas Selective Search costs ~2 s on the CPU.
8 (T/F). RoI pooling crops a region directly from the input image and resizes it to a fixed pixel size.
AnswerFalse. RoI pooling operates on the feature map (after the backbone CNN). It projects the proposal onto the feature map and pools that variable-size patch into a fixed-size grid (e.g. 7×7) so the FC head can consume it.
