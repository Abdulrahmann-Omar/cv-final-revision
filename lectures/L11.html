s11.1 — The complete Faster R-CNN pipeline

The Region-CNN family at a glance

Why it matters: every "two-stage" detector you will read about later (Mask R-CNN, Cascade R-CNN…) shares this skeleton. Owning the diagram makes everything else trivial.

Main idea of region-based CNNs: instead of brute-force sliding a window across the whole image, propose only a few promising regions and feed those to the detector. The detector returns, for every proposal, a refined bounding box $(x,y,w,h)$ and an object class.

The three generations sit on the same backbone:

  • R-CNN = Region proposal (Selective Search / Edge Boxes) + CNN run once per proposal. Slow.
  • Fast R-CNN = run the CNN once on the whole image, then project each proposal into the shared feature map and pool it.
  • Faster R-CNN = Fast R-CNN + a learned Region Proposal Network (RPN) that replaces Selective Search. Now everything is one trainable network.

Region-proposal techniques (recap)

Why it matters: the "front door" of R-CNN and Fast R-CNN; replaced by the RPN in Faster R-CNN. You need to recognise the names on the exam.

  • Selective Search — bottom-up: group pixels by similarity in colour, intensity, texture, …
  • Edge Boxes — top-down: score boxes by how many edge "contours" they enclose.
  • RPN (Faster R-CNN, L10) — a small CNN slid over the feature map predicting objectness + box regression from $k$ anchors per location.

Faster R-CNN — the four shared blocks

Why it matters: the canonical exam diagram. Memorise the boxes and arrows; everything else (training, losses, RoI Pool, RoI Align) hangs off it.

  1. Backbone CNN (pre-trained): input image (e.g. 224×224 or 800×576) → conv + pool stack → feature map. Typical backbone has 5 pooling layers, so spatial size is divided by $2^5 = 32$.
  2. Region Proposal Network (RPN): slides a small conv on the feature map; for every spatial location and every anchor outputs (a) objectness (object / background) and (b) box regression deltas. After non-max suppression you get a fixed-size list of proposals.
  3. RoI Pool / RoI Align: each proposal box is projected into the shared feature map; the variable-size patch is converted to a fixed $h\times w$ tensor (e.g. 7×7×512 or 3×3×512). This is the bridge that lets variable-size proposals feed a fixed-size FC.
  4. Detector head (Fast R-CNN head): FC layers → softmax over $(K+1)$ classes (K classes + background) and bbox refinement $(d_x,d_y,d_w,d_h)$ for each class.

Feature-map downscale — the "÷32" rule

After 5 pooling layers, a $224\times 224$ patch is represented by $224/32 = 7$ activations in each direction → $7\times 7\times 512$.

Equivalently, each value in the final feature map corresponds to a 32×32 region in the input image.

Common exam traps

  • "Faster R-CNN runs the CNN once per proposal." — False. The CNN runs once on the whole image; proposals are projected into the shared feature map. That was R-CNN.
  • "RPN replaces the detector." — False. RPN replaces Selective Search; the detector head (classifier + bbox regressor) is still there.
  • "RPN uses a different backbone from the detector." — False. They share the same backbone feature map (slide 12).
Q. List the four functional blocks of Faster R-CNN in order.
AnswerBackbone CNN → RPN (objectness + box regression on anchors) → RoI Pool / RoI Align → Detector head (softmax $K{+}1$ classes + per-class bbox refinement).
Q. Why is the feature map shared between the RPN and the detector?
AnswerComputing CNN features is the most expensive step. Running it once and reusing the same feature map for both heads saves time and lets the two heads learn complementary representations.

s11.2 — RoI Pooling (with quantisation)

What problem does RoI Pooling solve?

Why it matters: FC layers demand a fixed input size, but proposals come in arbitrary sizes/aspect ratios. RoI Pool/Align is the bridge that maps any variable-size projected RoI to a fixed $h\times w \times C$ tensor.

Input: a variable-size feature-map patch (the projected RoI).
Output: a fixed-size feature map, typically $7\times 7\times C$ (often $3\times 3$ in lecture toy examples).

Step 1 of RoI Pooling — quantise the box origin & size

Why it matters: this is the first place where information is silently lost. Recognising it is half the comparison with RoI Align on the exam.

The RPN/Selective-Search returns a proposal in image-space: $(x,y,W,H)$ in pixels. After projection into the feature map we divide by $32$ (for a 5-pool backbone), producing fractional coordinates. RoI Pool rounds them down (floor) so the patch aligns with integer feature-map cells.

Step 2 of RoI Pooling — quantise the bin boundaries

Why it matters: even after step 1 you must split the integer patch into $h \times w$ bins. If $H/h$ is not integral, the bin boundaries themselves get rounded → another loss.

The output grid is fixed (say $2\times 2$). Each bin is the feature-map patch of size $H/h \times W/w$. If that ratio is integer, no problem. If not, RoI Pool rounds each bin's size or origin → bins of unequal size, and parts of the patch are repeated or missed.

Worked example A — RoI Pooling, simple case

Feature map 25×25. Proposal in feature-map space is $14\times14$ starting near integer coords; we want a $7\times7$ output.

Step-by-step
  1. The proposal originally landed at (5.6, 7.2) on the FM, size 14.5×14.5. Step-1 rounding: floor origin to (6, 7); round size to 14 → the patch is the integer block from (6,7) to (20,21), i.e. exactly 14×14.
  2. Step-2 binning: target output is 7×7. Bin size = 14/7 = 2 → every bin is exactly 2×2. No quantisation error in step 2.
  3. Inside each 2×2 bin take the max of the 4 values → one number → write to the 7×7 output.
Information loss came only from step 1 (shift & size rounding); step 2 was clean because 14 is divisible by 7.

Worked example B — RoI Pooling, the ugly case

Same 14×14 patch but now we want a $3\times 3$ output instead of $7\times 7$.

Where it goes wrong Bin size would be $14/3 = 4.6\overline{6}$. RoI Pool rounds the bin boundaries — typical implementation floors them, so two bins become 4 wide and one bin becomes 6 wide (or some similar split). The "extra" rows/columns inside the wider bins are repeated or simply dropped in the output → genuine information loss.

This is exactly the situation on slides 20 and 47: width 7 split into 3 bins → grid width $7/3 \approx 2.33$ gets floored to 2. Two columns are "missing" from the output and one column is "added" from an adjacent strip.

RoI Pooling — algorithm in two lines

  1. Quantise proposal in feature-map coordinates: $\text{floor}(x/32),\ \text{floor}(y/32),\ \text{round}(W/32),\ \text{round}(H/32)$.
  2. Quantise bins: divide the integer patch into $h\times w$ bins by flooring the bin boundaries; for each bin take the max-pool across its FM values; repeat per channel.

The whole thing is two consecutive roundings — "double quantisation".

Common exam traps

  • "RoI Pool quantises only the box origin." — Wrong. It quantises twice: (1) the proposal box on the feature map, (2) the bin boundaries inside it.
  • "If $H/h$ is integer there is no error." — Step 2 is clean, but step 1 (the projection rounding) still loses information.
  • "Max pooling is applied across channels." — No, per channel. The output has the same number of channels as the input feature map.
Q. A proposal lands at (5.6, 7.2) in FM space, size 14.5×14.5; the target output is 7×7. State both quantisation steps and indicate which one is clean here.
AnswerStep 1 (origin/size): floors to (6,7) and 14×14 → loses sub-pixel info. Step 2 (binning): 14/7=2 exactly → clean.
Q. Same proposal but target 3×3. What does RoI Pool do with the bin width 14/3 = 4.67?
AnswerRounds the bin boundaries; some bins end up wider than others → some rows/columns are repeated or dropped → second source of information loss.

s11.3 — RoI Align (no quantisation, bilinear interpolation)

What RoI Align changes

Why it matters: RoI Align fixed the misalignment that hurt Fast/Faster R-CNN on small objects and was the secret sauce of Mask R-CNN. Comparing the two is the highest-yield exam question of this lecture.

Key idea: do not quantise anywhere. Keep the proposal coordinates as floats, split the patch into equal-size bins (also floats), sample a small fixed number of points inside each bin, and read off the feature-map value at each sample point using bilinear interpolation of the four neighbouring integer cells. Finally pool (max or average) over the samples to get one number per bin per channel.

Bilinear interpolation — the only formula you must memorise

Given the four integer neighbours $f(x_1,y_1)$, $f(x_2,y_1)$, $f(x_1,y_2)$, $f(x_2,y_2)$ (with $x_2 = x_1{+}1$, $y_2 = y_1{+}1$), the value at a fractional sample $(x,y)$ is

$$f(x,y) \approx (1-a)(1-b)\,f(x_1,y_1) + a(1-b)\,f(x_2,y_1) + (1-a)\,b\,f(x_1,y_2) + a\,b\,f(x_2,y_2),$$

with $a = x - x_1$ (fractional offset in x) and $b = y - y_1$ (fractional offset in y).

Equivalent two-step formulation (used in the slides):

  1. Horizontal interpolation on the top edge and bottom edge separately:
    $f_{\text{top}}(x) = f(x_1,y_1)(x_2-x) + f(x_2,y_1)(x-x_1)$
    $f_{\text{bot}}(x) = f(x_1,y_2)(x_2-x) + f(x_2,y_2)(x-x_1)$
  2. Vertical interpolation between the two:
    $f(x,y) = f_{\text{top}}(x)(y_2-y) + f_{\text{bot}}(x)(y-y_1)$
Slide 36 worked example by hand Corners: $f(3,5)=15$, $f(9,5)=35$, $f(3,8)=40$, $f(9,8)=20$. Sample point $(7,6)$.
Horizontal (y=5): $35\cdot\tfrac{4}{6} + 15\cdot\tfrac{2}{6} = 28.3$.
Horizontal (y=8): $20\cdot\tfrac{4}{6} + 40\cdot\tfrac{2}{6} = 26.6$.
Vertical (x=7): $28.3\cdot\tfrac{2}{3} + 26.6\cdot\tfrac{1}{3} = \boxed{27.7}$.

Worked example — RoI Align on the cow/person image

Why it matters: this is the lecture's signature numerical example; expect it back, possibly with different numbers.

Setup (slide 24–25): image 800×576, ground-truth box for the cow at $(x,y,W,H) = (300, 210, 230, 180)$. The backbone has 5 pool layers → divide every dimension by 32.

Feature-map coordinates:

  • Origin: $(300/32, 210/32) = (9.375, 6.563)$.
  • Width: $230/32 = 7.188$. Height: $180/32 = 5.625$.
  • Bottom-right corner: $(9.375 + 7.188, 6.563 + 5.625) = (16.563, 12.188)$.

Target output: $3 \times 3$. Cell size = $(7.188/3, 5.625/3) = (2.396, 1.875)$. No rounding.

Sample points (4 per cell, 2×2 grid inside): for cell with top-left $(X_c,Y_c)$ and cell size $(w_c, h_c) = (2.396, 1.875)$ the four samples are at offsets $\bigl(w_c\cdot\tfrac{1}{3}, h_c\cdot\tfrac{1}{3}\bigr)$, $\bigl(w_c\cdot\tfrac{2}{3}, h_c\cdot\tfrac{1}{3}\bigr)$, $\bigl(w_c\cdot\tfrac{1}{3}, h_c\cdot\tfrac{2}{3}\bigr)$, $\bigl(w_c\cdot\tfrac{2}{3}, h_c\cdot\tfrac{2}{3}\bigr)$.

First cell — all four samples computed Top-left of first cell: $(9.375, 6.563)$. Cell size $(2.396, 1.875)$. Sample points:
  • $S_1 = (9.375 + 2.396\cdot\tfrac{1}{3},\ 6.563 + 1.875\cdot\tfrac{1}{3}) = (10.17, 7.19)$
  • $S_2 = (9.375 + 2.396\cdot\tfrac{2}{3},\ 6.563 + 1.875\cdot\tfrac{1}{3}) = (10.97, 7.19)$
  • $S_3 = (9.375 + 2.396\cdot\tfrac{1}{3},\ 6.563 + 1.875\cdot\tfrac{2}{3}) = (10.17, 7.81)$
  • $S_4 = (9.375 + 2.396\cdot\tfrac{2}{3},\ 6.563 + 1.875\cdot\tfrac{2}{3}) = (10.97, 7.81)$
For $S_1 = (10.17, 7.19)$ the four neighbours on the FM are $f(9.5,6.5)=0.9$, $f(10.5,6.5)=0.8$, $f(9.5,7.5)=0.6$, $f(10.5,7.5)=0.2$ (using cell centres; slides 37–40):
Horizontal at y=6.5: $0.9\cdot(10.5-10.17) + 0.8\cdot(10.17-9.5) = 0.833$.
Horizontal at y=7.5: $0.6\cdot(10.5-10.17) + 0.2\cdot(10.17-9.5) = 0.332$.
Vertical at x=10.17: $0.833\cdot(7.5-7.19) + 0.332\cdot(7.19-6.5) = \boxed{0.487}$.

Similarly (slides 41–43): $S_2 = 0.436$, $S_3 = 0.332$, $S_4 = 0.281$.

Pool the 4 samples (max — what the slides do; some implementations average):

$\max(0.487,\ 0.436,\ 0.332,\ 0.281) = 0.487$.

Common exam traps

  • "RoI Align uses nearest-neighbour interpolation." — False. It uses bilinear.
  • "RoI Align needs lots of samples per cell to work." — False; the original paper uses just 4. The wins come from removing quantisation, not from extra samples.
  • "RoI Align is faster than RoI Pool." — False; it does more arithmetic. Its advantage is accuracy.
  • "RoI Align eliminates all rounding errors in detection." — False; it only fixes the pooling step. The RPN's anchor quantisation and the final box decoding still have their own approximations.
Q. Two sources of information loss in RoI Pooling and what RoI Align does about each.
Answer(1) Quantising the proposal's origin + size on the FM — RoI Align keeps them as floats. (2) Quantising the bin boundaries inside the proposal — RoI Align keeps the bins equal-size floats and reads values via bilinear interpolation at sample points.
Q (numeric). Bilinear: $f(3,5)=10$, $f(4,5)=20$, $f(3,6)=30$, $f(4,6)=40$. What is $f(3.25, 5.5)$?
Answer$a=0.25, b=0.5$. $f = 0.75\cdot0.5\cdot10 + 0.25\cdot0.5\cdot20 + 0.75\cdot0.5\cdot30 + 0.25\cdot0.5\cdot40 = 3.75 + 2.5 + 11.25 + 5 = \mathbf{22.5}$.
Q. In the lecture's worked example, why is the cell size $(2.396, 1.875)$ rather than integer?
AnswerBecause the projected RoI is $7.188\times 5.625$ feature-map cells and the target is $3\times 3$. RoI Align does not round these numbers, so the bins keep the exact float size $7.188/3 = 2.396$, $5.625/3 = 1.875$.

s11.4 — Detector head & training

Architecture of the Detector head (Fast / Faster R-CNN)

Why it matters: this is the "final mile" — what turns a pooled feature tensor into a class label and a refined box. The output dimensionalities are a classic exam multiplication question.

  • Input: one fixed-size RoI tensor (e.g. $3\times3\times512 = 4608$ values) from RoI Pool/Align.
  • Hidden: a stack of fully-connected layers (in the slide's ImageNet example, 3 hidden FCs).
  • Two output heads:
    • Classifier: Softmax over $K+1$ classes (K object classes + 1 background). Example: 1000 classes + bg = 1001 outputs.
    • Regressor: 4 box-refinement values $(d_x, d_y, d_w, d_h)$ per class. Example: $1000\times 4 = \mathbf{4000}$ outputs.

Box refinement — what those four numbers mean

The regressor does not predict an absolute box. It predicts offsets relative to the proposal:

$$\hat{x} = x_p + w_p\,d_x,\qquad \hat{y} = y_p + h_p\,d_y,\qquad \hat{w} = w_p\,e^{d_w},\qquad \hat{h} = h_p\,e^{d_h}.$$

So $(d_x, d_y)$ are normalised centre shifts and $(d_w, d_h)$ are log-scale size changes. Per class means the network learns class-specific shape priors (a person box prefers a different aspect ratio from a car box).

Loss function — multi-task

For each RoI the loss is a sum of two terms:

$$L = L_{\text{cls}}(p, c^*) + \lambda\,\mathbb{1}[c^* \neq \text{bg}]\,L_{\text{box}}(t, t^*),$$

  • $L_{\text{cls}}$ = softmax cross-entropy / log-loss over the $K+1$ class probabilities.
  • $L_{\text{box}}$ = Smooth-L1 (a.k.a. Huber) loss between predicted offsets $t = (d_x,d_y,d_w,d_h)$ of the assigned class and the ground-truth offsets $t^*$.
  • Indicator $\mathbb{1}[c^* \neq \text{bg}]$ → no box loss for background proposals (you can't "refine" a non-object).

Training Faster R-CNN — step-by-step

Why it matters: the training procedure has its own quirks (shared weights, IoU thresholds, two losses, 4-step alternating). All are exam-favourites.

  1. Run the RPN on a randomly-selected training image; apply non-max suppression and cap the number of proposals (slide 52).
  2. For each proposal, pick the corresponding (Regressor, Objectness) vector pair (slide 53).
  3. Label assignment (slide 54): compute IoU of the proposal with every ground-truth box.
    • If max IoU $< 0.5$ → label = Background.
    • Else label = class of the highest-IoU GT box, and the GT offsets become the regression target.
  4. Feature for the detector (slide 55): instead of cropping the image and rerunning the CNN (R-CNN style), project the proposal onto the shared feature map.
  5. Apply RoI Pool / RoI Align on that feature-map patch (slide 56).
  6. Feed to the FC detector → two output vectors (classification probs & per-class offsets) (slide 57).
  7. Apply the multi-task loss and back-prop (slide 58).
  8. Use shared weights: the same backbone CNN is used by both RPN and detector — never run the CNN on the same image twice (slides 59–60).

End-to-end training options

Why it matters: there are two ways to actually do the optimisation. The original Faster R-CNN paper proposes a 4-step alternating procedure; subsequent work uses end-to-end joint training. Both are fair game.

  • 4-step alternating training (original Ren et al. 2015):
    1. Train RPN with ImageNet-pretrained backbone.
    2. Train Fast R-CNN detector using the RPN's proposals — still its own backbone copy.
    3. Fix the shared conv layers, fine-tune only the RPN's RPN-specific layers (so the RPN learns to share the detector's features).
    4. Fix the shared conv layers again, fine-tune only the detector's detector-specific layers.
    Result: one unified network where RPN and detector share the same backbone.
  • Approximate joint / end-to-end training: the gradients from both heads back-propagate through the shared backbone in one optimiser step. The "approximate" is because the gradients of the RPN-produced proposal coordinates into RoI Pool are normally ignored. Simpler to implement and now standard.

Trap

  • "End-to-end training of Faster R-CNN was impossible." — False. End-to-end is the modern default; alternating was just the original recipe.
  • "The detector head computes box loss on background proposals." — False. Background contributes only to the classification loss.

Practice quiz — Lecture 11

1 (MCQ). Why is the same shared feature map used by both the RPN and the detector in Faster R-CNN?
a) Different conv weights for each head waste training time.   b) Computing the CNN features is the most expensive step; sharing saves time and lets both heads benefit from the same representation.   c) Because the RPN cannot use convolution.   d) Because the detector cannot use anchors.
Answerb. CNN features are expensive; running once and reusing avoids duplication, and the shared backbone learns features useful to both tasks.
2 (Comparison). Name the two quantisation steps in RoI Pooling that RoI Align eliminates, and how RoI Align reads feature values instead.
Answer(1) Quantisation of the proposal box origin/size on the feature map. (2) Quantisation of the bin boundaries inside the proposal. RoI Align keeps both as floats and uses bilinear interpolation at 4 (or 9) sample points per bin, then max- or avg-pools over the samples.
3 (Numeric, bilinear). The FM values at four neighbours are $f(9.5,6.5)=0.9$, $f(10.5,6.5)=0.8$, $f(9.5,7.5)=0.6$, $f(10.5,7.5)=0.2$. Find $f(10.17, 7.19)$ by bilinear interpolation.
AnswerHorizontal at y=6.5: $0.9\cdot(10.5-10.17)+0.8\cdot(10.17-9.5)= 0.833$. Horizontal at y=7.5: $0.6\cdot0.33+0.2\cdot0.67=0.332$. Vertical at x=10.17: $0.833\cdot(7.5-7.19)+0.332\cdot(7.19-6.5) = \mathbf{0.487}$.
4 (Numeric, RoI Pool). FM 25×25, proposal in FM space $14.5\times14.5$ starting at $(5.6, 7.2)$, target $7\times 7$. State both quantisation steps and explain why step 2 is clean here.
AnswerStep 1: floor origin to $(6,7)$, size to $14\times 14$. Step 2: bin size $14/7 = 2$ → exactly $2\times 2$ bins, no fractional split → step 2 is clean. Information loss came from step 1.
5 (Numeric, detector). Imagenet 1000 classes. How many outputs does the detector head produce in (i) the classifier and (ii) the regressor?
Answer(i) $K+1 = 1001$ softmax outputs (1000 classes + background). (ii) $4\cdot K = 4000$ regression outputs (4 offsets per class; background gets no box loss).
6 (T/F). When training, the box-regression loss is computed for every proposal, including background.
AnswerFalse. Background proposals contribute only to the classification log-loss; there is no GT box to regress to so the indicator $\mathbb{1}[c^* \neq \text{bg}]$ zeroes the box loss.
7 (Scenario). You train Faster R-CNN with RoI Pool and observe poor mAP on small objects. Suggest one architectural change that should help, and explain why.
AnswerReplace RoI Pool with RoI Align. Small objects span only a handful of FM cells, so the $\pm 1$-cell misalignment caused by RoI Pool's double quantisation is a large fraction of the object's footprint. Bilinear-interpolated sampling is sub-pixel accurate and recovers the lost localisation precision.
8 (Short). Outline the 4-step alternating training procedure for Faster R-CNN.
Answer(1) Train the RPN with an ImageNet-pretrained backbone. (2) Train the Fast R-CNN detector using the RPN's proposals (its own backbone copy). (3) Freeze the shared conv layers and fine-tune the RPN-specific layers on top of the detector's backbone. (4) Freeze the shared conv layers again and fine-tune only the detector-specific layers. Result: one unified network with a shared backbone.
