s12.1 — Detector Training (Multi-task loss on RoIs)

The pipeline you are training

Why it matters: L11 built the Faster R-CNN architecture; here we have to put numbers on the loss and tell the optimiser which RoIs are positive examples, which are negative, and how to mix them in a mini-batch.

Recall the data-flow per training image:

  1. Run the shared CNN backbone → produce a feature map.
  2. Run RPN + NMS + max-count → get a small set of region proposals (each one is a 4-D box vector + an objectness score).
  3. For each proposal: project the box onto the feature map, apply RoI (Align) Pooling, feed the fixed-size tensor through the Fast R-CNN detector head, get two outputs — a class-probability vector ($N\!+\!1$ classes including background) and a per-class regressor vector ($4N$ offsets).
  4. Compute the multi-task loss against ground-truth labels (classes & refined box offsets) and back-propagate.

Labelling a proposal: positive (object), negative (background), or mix

Why it matters: an RPN proposal is not annotated; we must derive its ground-truth label from the original GT boxes using IoU. This is the rule the slides will quiz you on.

For every proposal compare its IoU against every ground-truth box:

  • If max IoU $< 0.5$ → label the proposal as Background ($C_{bg}$). Its target class one-hot is $[0,0,\ldots,0,1]$, and there is no bounding-box target.
  • Else → match the proposal to the GT box that gave the highest IoU. Copy that GT box's class (one-hot at $C_u$) and compute the four refinement offsets $[d_x,d_y,d_w,d_h]$ relative to the proposal.

The detector then has two output heads:

  • Classifier head: softmax over $N+1$ classes (the $N$ object classes plus a background class $C_{bg}$).
  • Regressor head: $4N$ outputs — four offsets per object class. Only the four offsets for the GT class are used in the loss; the rest are "don't care".

The multi-task loss (the formula you will be examined on)

Why it matters: the Fast R-CNN multi-task loss is the equation that ties classification and box regression together. Every R-CNN successor (Mask R-CNN, etc.) extends it.

$$\boxed{\;L(p,u,t^u,v) \;=\; L_{cls}(p,u)\;+\;\lambda\,[\,u\ge 1\,]\,L_{loc}(t^u,v)\;}$$

Symbols (memorise):

  • $p$ — predicted class-probability vector over $N+1$ classes (softmax output).
  • $u$ — the ground-truth class index for this proposal. By convention $u=0$ means background; $u\ge 1$ means a real object class.
  • $t^u$ — the four predicted bounding-box offsets for class $u$ (the rest of the $4N$ outputs are discarded).
  • $v$ — the four ground-truth offsets $[d_x,d_y,d_w,d_h]$.
  • $\lambda$ — a balancing weight between the two losses (typically $\lambda=1$).
  • $[u\ge 1]$ — the Iverson bracket: equals $1$ when $u\ge 1$ (foreground) and $0$ when $u=0$ (background). This is exactly the slide's "No offsets error for Background class" rule.

The two pieces are:

  • $L_{cls}(p,u)=-\log p_u$ — standard softmax cross-entropy (the slide labels it "Log Loss"). It is evaluated over all $N+1$ class scores.
  • $L_{loc}(t^u,v)=\sum_{i\in\{x,y,w,h\}}\text{smooth}_{L_1}(t^u_i - v_i)$, where $$\text{smooth}_{L_1}(x)=\begin{cases}0.5\,x^2 & |x|<1\\ |x|-0.5 & \text{otherwise}\end{cases}$$ The slide draws it as "Squared Distance Loss" (L2-flavoured) — smooth-L1 is the robust version used in Fast R-CNN to avoid exploding gradients for outlier offsets.
Worked example — the Iverson bracket in action Two proposals at the same training step:
  • Proposal A — max IoU = 0.7 with a "car" (class index $u=7$). $L = L_{cls}(p,7) + \lambda\cdot 1 \cdot L_{loc}(t^7,v)$. Both terms contribute.
  • Proposal B — max IoU = 0.2, so labelled background ($u=0$). $L = L_{cls}(p,0) + \lambda\cdot 0 \cdot L_{loc}(\cdot) = L_{cls}(p,0)$. Only the class-log-loss contributes; the $4N$ regressor outputs receive zero gradient.

Common exam traps

  • "$L_{loc}$ is computed for background RoIs." — False. The Iverson bracket $[u\ge 1]$ zeros it out.
  • "$L_{loc}$ uses all $4N$ regressor outputs." — False. Only the 4 outputs that correspond to the GT class $u$ are penalised. The slide explicitly says "Loss is calculated for the 4 values of this object only, other values are assumed as Don't care".
  • "$L_{cls}$ uses only the GT-class probability." — Loss value reduces to $-\log p_u$, but the softmax denominator couples all $N+1$ logits, so gradient flows into every class score.
  • "Pure L2 (squared) loss is used." — In practice Fast R-CNN uses smooth-L1; on the slide it appears as "Squared Distance Loss" but in the paper it is the robust version.

Mini-batch composition (positive vs negative RoIs)

Why it matters: most RPN proposals are background; if you just sampled at random, the loss would be dominated by easy negatives. The R-CNN family fixes this by hard-coded sampling ratios.

HeadBatch sizePositivesNegativesImages / batch
RPN256128 (anchor IoU $\ge 0.7$)128 (anchor IoU $\le 0.3$)2 (may slow training)
Fast R-CNN detector + CNN12832 (RoI IoU $\ge 0.5$)96 (RoI IoU in $[0.1,0.5)$)2

Why the asymmetry? Negatives are abundant and cheap, so the detector mini-batch keeps a 3:1 negative-to-positive ratio — enough negatives to learn what background looks like, but not so many that positives are drowned out.

Q. A training image generates 300 proposals from the RPN. After IoU labelling, 12 are foreground and 288 are background. How does the detector training step build its mini-batch?
AnswerTake the 12 foreground RoIs (or randomly down-sample to 32 if more) plus enough random background RoIs to reach 96, for a total batch of 128. If foreground is short, pad with extra background. Two such images jointly fill the 128-RoI batch.
Q (T/F). The RPN batch and the detector batch use the same positive-to-negative ratio.
AnswerFalse. RPN uses 1:1 (128:128); the detector uses 1:3 (32:96).

Inference — what comes out at the end

Why it matters: training and inference share the architecture, but inference adds a final per-class NMS step on the detector outputs to drop duplicate boxes.

Two NMS passes in Faster R-CNN

  1. RPN NMS (class-agnostic) — dedup anchor proposals using objectness scores.
  2. Detector NMS (per-class) — dedup the final detections within each class.

s12.2 — The 4-Step Alternate Training Schedule

The problem and the recipe

Why it matters: the RPN and the detector each need their own gradients, but they want to share the convolutional backbone. The original Faster R-CNN paper proposes a 4-step alternating schedule that achieves shared features without joint optimisation.

Initial state: a CNN backbone (e.g. VGG / ResNet) pre-trained on ImageNet classification — used as a feature extractor.

StepWhat is trainedWhat is frozenBackbone shared?
1RPN head + its own CNN copy (init from ImageNet)—No (separate backbones)
2Fast R-CNN detector + its own CNN copy (init from ImageNet), using proposals from step-1 RPNThe step-1 RPNNo (separate backbones)
3RPN-specific layers (re-finetune on step-2 backbone)Shared conv layers (from step-2 detector)Yes — backbone now becomes shared
4Detector-specific layers (fine-tune)Shared conv layersYes — both heads share the same backbone

After step 4 the two heads sit on top of the same convolutional feature map → the deployed Faster R-CNN model.

Step 1 — Train the RPN from ImageNet

Use the RPN multi-task loss (objectness log-loss + smooth-L1 box-offset regression for positive anchors only). The CNN backbone here is the ImageNet copy.

Step 2 — Train the Fast R-CNN detector with the step-1 proposals

At this point we have two CNN backbones — one for the RPN, one for the detector — that drifted apart during steps 1 and 2.

Step 3 — Freeze shared conv layers; fine-tune the RPN-specific layers

Take the step-2 detector's CNN (now considered the "shared backbone") and freeze it. Re-train only the RPN-specific layers (the small conv + the cls/reg sibling layers) so they re-align to this frozen backbone. Now the RPN and the detector share their convs.

Step 4 — Freeze shared conv layers; fine-tune the detector-specific layers

Keep the shared backbone frozen. Re-finetune only the detector-specific layers (RoI head + classifier + regressor). After step 4, RPN and detector live on a unified, shared backbone — the deployed Faster R-CNN.

Quick mental diagram of what is unfrozen each step 
Step 1:  [CNN (ImageNet) | RPN]                 -- both unfrozen
Step 2:  [CNN (fresh ImageNet) | Detector]      -- both unfrozen (separate CNN)
Step 3:  [CNN (frozen, from step 2) | RPN]      -- only RPN-specific layers unfrozen
Step 4:  [CNN (frozen) | Detector]              -- only detector-specific layers unfrozen
        → backbone is now shared by RPN and Detector ✓
Q. In step 3, why is it valid to freeze the backbone instead of jointly training it with the RPN?
AnswerBecause step 2 has already adapted that backbone to be discriminative for the detection classes. Freezing it forces the RPN to adapt to the detector's features — this is precisely what creates a single shared backbone.

The modern alternative — joint end-to-end training

Why it matters: the alternating recipe is the original method; today's implementations usually replace it with a simpler scheme.

Sum all four losses (RPN cls + RPN reg + detector cls + detector reg) into a single objective and back-prop through one shared backbone in one training loop. This is faster to implement and trains the backbone with both signals simultaneously; the alternate schedule is still useful didactically because it makes the data dependency explicit.

Common exam traps

  • "All four steps unfreeze the backbone." — False. Steps 1 & 2 train it; steps 3 & 4 freeze it.
  • "After step 2 the backbone is already shared." — False. Steps 1 and 2 grow two separate backbones. Sharing happens at step 3 when we adopt step-2's backbone for the RPN too.
  • "Step 1 trains the RPN from scratch." — Misleading; the CNN backbone is initialised from ImageNet pre-training, then the RPN-specific layers are trained on top.

s12.3 — Mask R-CNN (instance segmentation)

Segmentation 101 — semantic vs instance

Why it matters: detection gives bounding boxes; segmentation classifies pixels. The two flavours differ in whether different instances of the same class are separated.

Fully Convolutional Network (FCN) — the semantic-segmentation baseline

Why it matters: Mask R-CNN's mask head is a small FCN attached to each RoI. Understanding FCN first makes the Mask R-CNN head trivial.

FCN training intuition Treat segmentation as per-pixel classification: at every output pixel, the network produces $C$ class scores; the loss is per-pixel softmax cross-entropy against the GT class at that pixel. Skip connections from earlier pooling layers (FCN-16s, FCN-8s) inject high-resolution information back into the prediction.

Mask R-CNN = Faster R-CNN + a parallel FCN mask head

Why it matters: this is the headline architecture of the lecture. It adds instance segmentation almost for free on top of a Faster R-CNN.

Key architectural changes vs Faster R-CNN:

  1. Replace RoI Pool (which quantises feature-map coordinates twice) with RoI Align (bilinear sampling, no quantisation). For boxes this only marginally helps; for masks it is essential because mask quality depends on pixel-accurate alignment.
  2. Add a third branch in parallel with cls/box: a small FCN that outputs $K$ binary masks of size $m\times m$ (e.g. $28\times 28$).

Decoupling class and mask — sigmoid per channel, not softmax

Why it matters: this is the single most important Mask R-CNN design decision and a favourite exam point.

Output of the mask head: a tensor of shape $K \times m \times m$ (e.g. $80 \times 28 \times 28$). Each of the $K$ channels is passed through a sigmoid pixel-wise — they are independent binary mask predictions, one per class.

Mask R-CNN loss

Why it matters: this is the exam formula for this lecture.

$$\boxed{\;L \;=\; L_{cls} \;+\; L_{box} \;+\; L_{mask}\;}$$

  • $L_{cls}$ — same softmax log-loss as Faster R-CNN, over $N+1$ classes.
  • $L_{box}$ — same smooth-L1 on the four box offsets of the GT class (Iverson bracket on background still applies).
  • $L_{mask}$ — average per-pixel binary cross-entropy across an $m\times m$ mask, evaluated only on the mask channel of the GT class $u$: $$L_{mask}\;=\;-\frac{1}{m^2}\sum_{(i,j)}\Big[y_{ij}\log\hat{y}^{(u)}_{ij} + (1-y_{ij})\log(1-\hat{y}^{(u)}_{ij})\Big]$$ where $\hat{y}^{(u)}_{ij}$ is the sigmoid prediction at pixel $(i,j)$ of the $u$-th mask channel, and $y_{ij}\in\{0,1\}$ is the binary GT mask at that pixel (1 inside the object, 0 outside).

The mask loss is decoupled from class prediction: the other $K-1$ channels receive no gradient from $L_{mask}$ on this RoI. This is why sigmoid + per-class channels works — classes never compete inside the mask head.

Worked example — 256×256 image with one car (class 7 of 80) Single RoI for the car. Pipeline:
  1. RoI Align extracts a $14\times 14\times 256$ feature for this RoI.
  2. The mask head (4 convs at 14×14, one transposed conv to 28×28, final 1×1 conv with 80 channels) outputs a tensor of shape $80\times 28\times 28$.
  3. Apply sigmoid to every pixel of every channel.
  4. During training: take only channel $u=7$ (the GT class) and compute per-pixel binary CE against the GT mask resized to $28\times 28$. The other 79 channels are ignored for $L_{mask}$ on this RoI.
  5. $L_{cls}$ is still computed over all 81 class scores; $L_{box}$ is computed over the 4 offsets of class 7.
  6. Total loss: $L = L_{cls} + L_{box} + L_{mask}^{(7)}$.

From 28×28 soft mask to image-resolution binary mask

Why it matters: at inference time the 28×28 sigmoid output has to be projected back to image coordinates.

Steps (inference):

  1. Class head picks the predicted class $\hat u$ for this RoI (and a confidence).
  2. Box head gives the refined $(x,y,w,h)$.
  3. Take the $\hat u$-th channel of the $K\times 28\times 28$ mask tensor. Apply sigmoid → values in $(0,1)$.
  4. Bilinear-resample the $28\times 28$ soft mask to the size of the predicted bounding box (in image pixels).
  5. Threshold at $0.5$ → final binary instance mask (foreground vs background).

Why RoI Align matters for masks (not just bbox)

Why it matters: the slides keep noting "ROI (& Align) Pooling". For masks, RoI Align is no longer optional.

RoI Pool quantises the proposal twice (when mapping image → feature coords, and when bucketing into the output grid). For a 16× stride, a misalignment of ~half a feature-map cell = ~8 image pixels. For boxes you barely notice; for pixel-level masks this kills accuracy.

RoI Align keeps the floating-point coordinates and samples the feature map with bilinear interpolation at 4 regularly-spaced points per output cell, then averages. No quantisation. This is what makes pixel-accurate instance masks possible.

Common exam traps

  • "Mask R-CNN replaces softmax with sigmoid for classification." — False. The class head is still softmax. The change is in the mask head: per-class sigmoid instead of cross-class softmax.
  • "$L_{mask}$ uses all $K$ channels." — False. Only the channel of the GT class contributes.
  • "$L_{mask}$ is per-pixel softmax cross-entropy." — That is FCN, not Mask R-CNN. Mask R-CNN is per-pixel binary CE.
  • "RoI Pool is fine for Mask R-CNN." — Empirically no; pixel-level alignment matters and RoI Align gives ~1–3 mAP improvement on masks.
  • Confusing the mask output spatial size: 28×28 with the FPN backbone (slide 34) vs 14×14 with the C5/ResNet backbone (slide 35).

Practice quiz — Lecture 12

1 (Short). Write the Fast R-CNN multi-task loss and name every symbol.
Answer$L(p,u,t^u,v)=L_{cls}(p,u)+\lambda[u\ge 1]L_{loc}(t^u,v)$. $p$ = predicted class probabilities (softmax), $u$ = GT class index (0 = background), $t^u$ = predicted box offsets for class $u$, $v$ = ground-truth offsets, $\lambda$ = balancing weight, $[u\ge 1]$ = Iverson bracket that zeros the regression term for background RoIs.
2 (T/F). For a background RoI ($u=0$), the regression loss $L_{loc}$ contributes a non-zero value.
AnswerFalse. The Iverson bracket $[u\ge 1]$ is 0 for $u=0$, so $L_{loc}$ is multiplied by 0.
3 (List). State the 4 steps of the alternate training schedule.
Answer
  1. Train RPN, initialised from ImageNet pre-training.
  2. Train Fast R-CNN detector using proposals from step-1 RPN — using a separate ImageNet-initialised CNN (so far: two separate backbones).
  3. Freeze the shared conv layers (step-2's CNN); fine-tune RPN-specific layers on it (RPN and detector now share the backbone).
  4. Freeze the shared conv layers; fine-tune detector-specific layers. Done — both heads share one backbone.
4 (MCQ). The detector mini-batch has size 128. How is it composed?
a) 64 +, 64 −   b) 32 +, 96 −   c) 96 +, 32 −   d) 128 + only
Answerb) 32 positive + 96 negative (≈ 1:3), sampled from two images per batch.
5 (Short). Write the Mask R-CNN loss and explain what $L_{mask}$ measures.
Answer$L = L_{cls} + L_{box} + L_{mask}$. $L_{mask}$ is the average per-pixel binary cross-entropy over the $m\times m$ output mask, evaluated only on the channel corresponding to the ground-truth class $u$. The other $K-1$ mask channels are ignored — masks are decoupled from class prediction.
6 (T/F). RoI Pool and RoI Align give similar quality for Mask R-CNN's mask outputs.
AnswerFalse. RoI Pool quantises feature-map coordinates twice, which destroys pixel-level alignment. RoI Align uses bilinear sampling without quantisation and is essential for mask quality; it gives a sizeable mAP improvement on the mask metric.
7 (Numerical / scenario). A 256×256 image with one car (class index 7, COCO with 80 classes). The mask head produces an $80\times 28\times 28$ tensor for the car RoI. During training, which channel(s) of this tensor contribute to $L_{mask}$, and what activation/loss are applied?
AnswerOnly channel 7 (the GT-class channel). A sigmoid is applied pixel-wise to that 28×28 channel; the loss is the average binary cross-entropy between the 784 sigmoid outputs and the GT mask resized to 28×28. The other 79 channels are "don't care" for $L_{mask}$ (they still get gradients from other RoIs whose GT classes happen to fall on those channels).
8 (Short). Why does Mask R-CNN use a sigmoid per channel instead of a softmax across the 80 channels (as FCN does)?
AnswerTo decouple mask prediction from class prediction. With softmax across channels, the K masks compete pixel-by-pixel, forcing the network to also implicitly classify. With sigmoid per channel, the K masks are independent binary masks; the class head alone decides which channel to keep. Empirically this gives +5.5 mAP on COCO (24.8 → 30.3).
9 (T/F). Inside Faster R-CNN, NMS is applied exactly once during inference.
AnswerFalse. NMS is applied twice: once class-agnostic on RPN proposals (to dedup anchors), and again per-class on the final detector outputs (to dedup detections within each class).
