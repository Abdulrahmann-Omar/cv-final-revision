Question 1 · Part A — Mean Shift for image segmentation

i. Main steps of Mean Shift segmentation

Mean Shift treats every pixel as a point in feature space (e.g. (x, y, R, G, B)), drops a window on each point, and keeps sliding that window uphill toward the densest region. Points that climb to the same peak get the same label. The one knob you set is the window (bandwidth) size.
Step 1. Put every pixel into a feature vector — usually 5-D: (x, y, R, G, B). This is the "feature space" the algorithm searches in.
Step 2. Around each point, place a window of the chosen bandwidth. Compute the mean of all points inside that window.
Step 3. Move the window so its centre lands on that mean. This is the "mean shift" step — it nudges the window toward the locally densest area.
Step 4. Repeat Step 2/3 until the window stops moving (a mode of the density). Tag the starting pixel with the mode it landed on.
Step 5. Pixels that converge to the same mode form one segment. That's the segmentation.
Parameter to set: the window (kernel) size / bandwidth — this is the single most important knob.

ii. Sensitivity to the window size

Big window = looks at a lot of pixels at once → fewer steps to converge, but the mode it lands on is fuzzy (over-smooths, merges segments). Small window = sees fine detail and stays accurate, but each pixel walks slowly and the algorithm takes much longer.
Window sizeConvergence timeAccuracy / detail
Too bigFast (low complexity)Low — over-smooths, merges segments
Too smallSlow (high complexity)High — preserves detail, but may over-segment
Sweet spot: just big enough to span one homogeneous region, but no bigger.

Question 1 · Part B — Fit a line with RANSAC

i. Using RANSAC to fit a line to a noisy 2-D point cloud

RANSAC = "guess a model from a tiny random sample, count how many points agree with it, repeat, keep the best". For a line you only need 2 points to define a candidate. Outliers can't all line up by accident, so the winning model is the one most real points support.
Step 1. Pick 2 random points from the cloud and fit the line that passes through them — that's your candidate model.
Step 2. Check every other point: if its perpendicular distance to the line is below a small threshold t, it's an inlier, otherwise it's an outlier.
Step 3. Score this candidate by the number of inliers (the "consensus set").
Step 4. Repeat N times with fresh random pairs. Keep the candidate with the largest consensus set.
Step 5. Re-fit the line using least squares on the inliers of the winning candidate. That's your final line.

ii. Minimum iterations N for 60% inliers, 95% confidence

Use the standard RANSAC formula: $$N \;\geq\; \frac{\log(1-p)}{\log\!\big(1-(1-e)^{s}\big)}$$ where p = wanted probability of one clean sample, e = outlier ratio, s = sample size (here 2 for a line).
Plug in: p = 0.95, outlier ratio e = 1 − 0.6 = 0.4, s = 2.
Inner term: $(1-e)^{s} = 0.6^{2} = 0.36$, so $1 - 0.36 = 0.64$.
$N \geq \dfrac{\log(0.05)}{\log(0.64)} = \dfrac{-1.301}{-0.194} \approx 6.71$.
Round up: N = 7 iterations are enough to hit a clean (outlier-free) pair with 95% probability.

Question 2 · Part A — Which detector for rotation + zoom? Why is SIFT rotation-invariant?

Detector to use: SIFT

Whenever a question says "different rotation and zoom", you need a detector that is both scale-invariant and rotation-invariant. SIFT is built specifically for that combo: scale via DoG pyramid, rotation via a canonical orientation per keypoint.
Answer: SIFT (Scale-Invariant Feature Transform) — robust to scale, rotation, and moderate illumination/viewpoint change.

How SIFT achieves rotation invariance (the steps that matter)

Step 1 — Around each keypoint, compute image gradients. In a small window around the keypoint, compute the gradient magnitude and orientation at every pixel (weighted by a Gaussian centred on the keypoint).
Step 2 — Build an orientation histogram. Bin the orientations into 36 bins of 10° each. Each pixel votes into its bin, weighted by its gradient magnitude. The result is a 36-bin histogram saying "which direction is dominant here?".
Step 3 — Pick the dominant orientation. The tallest bin in that histogram is the keypoint's canonical orientation θ. (Any bin within 80% of the peak gets its own duplicate keypoint — that's why some keypoints have multiple orientations.)
Step 4 — Rotate the descriptor frame by −θ. Before computing the 128-D descriptor, every gradient direction is rotated so the dominant direction lines up with "up". From then on, the descriptor lives in a coordinate frame attached to the feature itself, not to the image axes.
Why this gives rotation invariance: rotate the image by any angle α — the keypoint's gradients all rotate by α too, so its dominant orientation also rotates by α. Subtracting θ wipes out that rotation, leaving the descriptor unchanged. Two photos of the same patch taken at different rotations end up with the same 128-D vector.

Question 2 · Part B — Skin-colour recognition with a Bayesian classifier

i. Low-level feature to extract

"Skin" is a colour cue, not a shape cue. RGB mixes brightness with colour, so it's a bad feature when lighting varies. The Hue channel of HSV isolates pure colour from intensity — exactly what you want for skin detection.
Use the Hue / colour value (e.g. the H channel of HSV) of each pixel as the feature x. Skin has a fairly narrow hue range, mostly independent of how brightly the scene is lit.

ii. Plugging in expert knowledge with a Bayesian classifier

Bayes lets you combine "what does skin look like?" (the likelihood) with "how often does skin appear?" (the prior). The prior is exactly where you bake in expert knowledge such as "in face photos, ~20% of pixels are skin".
For a pixel with feature x and a class Ck ∈ {skin, not-skin}, Bayes gives: $$P(C_k\mid x) \;=\; \frac{P(x\mid C_k)\,P(C_k)}{P(x)}$$
P(x | Ck) — the likelihood. Learnt from training pixels — typically a 1-D histogram or a small Gaussian over hue, fitted on a labelled skin dataset.
P(Ck) — the prior. This is the slot for expert knowledge: set P(skin) to the rough fraction of skin pixels you expect (e.g. 0.2 for face photos, 0.05 for street scenes). Larger prior → classifier is more willing to call a pixel skin.
Decision rule. Pixel is "skin" if P(skin | x) > P(not-skin | x). Because P(x) is the same in both, you can drop it and just compare P(x | Ck)·P(Ck).
In short: likelihood is learnt from data, prior carries the expert knowledge about skin frequency. Multiply them; pick the bigger one.

Question 2 · Part C — AdaBoost on a 1-D 2-class problem

The setup, in plain English

All 8 points lie on the x-axis (the y-coordinate is irrelevant). Reading their x-values from left to right:
  • x = −3, −1, 1, 3 (top row): classes are −1, +1, +1, −1.
  • x = −3, −1, 1, 3 (bottom row): same pattern.
So along the axis: (−, +, +, −) repeated. A single vertical line ("is x < threshold?") can never separate this pattern — the classes alternate. Classic XOR-like layout.

i. First weak learner + weight update

Initial weights. Every point starts equal: wi = 1/8 = 0.125.
Pick the best weak learner. A weak learner here is a vertical decision stump "predict +1 if x > θ, else −1" (or the reverse). The most "balanced" placement is the one in the middle (e.g. θ = 0). With that boundary, exactly 4 of the 8 points are misclassified — the two outer-left and outer-right ones still match, but the inner two pairs don't.
Weighted error. $\text{err} = \dfrac{\sum_i w_i \cdot \mathbf{1}[y_i \neq g_m(x_i)]}{\sum_i w_i} = \dfrac{4 \times 0.125}{1} = \mathbf{0.5}$.
Update rule for the misclassified points: $$w_i^{(m)} = w_i^{(m-1)} \cdot \frac{1-\text{err}}{\text{err}} = 0.125 \cdot \frac{1-0.5}{0.5} = 0.125.$$ Correctly-classified points keep their old weight (also 0.125). So every weight stays at 0.125 — nothing changed.
First weak learner: vertical line in the middle. err = 0.5. Update: every weight remains 0.125 — AdaBoost is stuck.

ii. Is one weak learner enough? What's needed?

A weak learner's voting weight is α = ½ ln((1 − err)/err). When err = 0.5, α = 0 — that classifier contributes nothing. And because all sample weights stayed equal, the next iteration will pick the same useless boundary forever.
No, one weak learner is not enough. err = 0.5 means the stump is no better than random, so its α = 0 and the final classifier is identically 0.

What's needed: at least three weak learners — typically two stumps that split the line into the three regions {x < −2}, {−2 ≤ x ≤ 2}, {x > 2}, plus a third that acts as a bias term to flip the middle region's sign. Equivalently: lift the data into 2-D (e.g. add x², or use a non-linear weak learner) so a linear stump can actually separate the alternating pattern.
