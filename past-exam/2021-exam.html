Midterm · Question 1·A — SIFT: scale space, DoG, orientation, normalisation

Solution — the four SIFT building blocks

Memorise SIFT as a four-step pipeline: blur stack → find blobs across scale → rotate the patch → describe it. The four items in the question map to these four pieces in order.

i. Scale space

A stack of progressively blurred versions of the image (Gaussian with increasing σ), arranged in octaves (each octave is half the resolution of the one above). It lets the same keypoint be found whether the object is photographed close-up or far away.
Purpose: represent the image at multiple scales so we can find features that are invariant to image scale (zoom in / zoom out).

ii. Difference of Gaussian (DoG)

Inside each octave, subtract neighbouring blurred images: DoG(σ) = G(kσ) − G(σ). The result approximates the scale-normalised Laplacian-of-Gaussian at that σ.
Purpose: a cheap approximation of the Laplacian used to detect blob-like keypoints as local extrema in 3D (x, y, scale). Less computation than computing LoG directly.

iii. Orientation assignment

Around each keypoint, build a histogram of gradient orientations weighted by gradient magnitude and a Gaussian window. The peak of the histogram becomes the keypoint's canonical direction.
Purpose: gives the keypoint rotation invariance. Everything described after this step is measured relative to that canonical direction, so a rotated copy of the same patch produces the same descriptor.

iv. Normalisation of the window's rotation and scale

Take the local 16×16 window, rotate it so the dominant direction points "up" and resize it to a fixed size relative to the keypoint's scale. Only then is the gradient histogram (the 128-D descriptor) computed.
Purpose: aligns every patch into the same canonical scale and orientation before description, so two views of the same point produce nearly identical descriptors.

Midterm · Question 1·B — Siamese Network with triplet loss (face verification)

Architecture

A Siamese network for verification is really one CNN used three times in parallel (shared weights) that produces an embedding f(x). Training pulls embeddings of the same person together and pushes embeddings of different people apart.
Sketch: three image inputs feed three copies of the same CNN (weights tied) → each output is a feature embedding f(x) ∈ ℝd. Distances are L2 distances between embeddings.
  • Anchor image A → f(A)
  • Positive image P (same person as A) → f(P)
  • Negative image N (different person) → f(N)
For two images of the same person, ‖f(x⁽¹⁾) − f(x⁽²⁾)‖² is small; for two images of different people, it is large.

Training procedure (the three ingredients)

1. Input

Triplets (A, P, N). Each triplet is: an anchor image, a positive image of the same class, and a negative image of a different class.

2. Learning objective

Minimise the distance between the anchor and the positive, and at the same time maximise the distance between the anchor and the negative — by at least a margin α.

3. Loss function — Triplet loss

$$\mathcal{L}(A, P, N) = \max\!\big(\, \|f(A) - f(P)\|_2^2 \;-\; \|f(A) - f(N)\|_2^2 \;+\; \alpha,\; 0 \,\big)$$ The max(·, 0) means: once the negative is already further than the positive by margin α, the triplet contributes zero loss (it's already "easy"). Training focuses on the hard / semi-hard triplets where the margin is not yet satisfied.
At test time you compare f(query) with each enrolled f(template) — if the distance is below a threshold, the two faces are declared the same person.

Final · Question 3 — Optical flow & Bayes background modelling (17 marks)

Part A·i — Algorithm & constraints (NOT covered in our current syllabus)

Per the official answer key, Question 3·A is marked "مش علينا" — outside our 2025/2026 scope. The classical answer would describe Lucas–Kanade optical flow under brightness constancy and small-motion assumptions, solved per local window in least-squares form.

Part A·ii — Big shifts (≥ 1 pixel) and the iterative resolution

The "small-motion" assumption (one-pixel Taylor expansion) breaks for large displacements. The fix is the pyramidal (coarse-to-fine) Lucas–Kanade: build a Gaussian pyramid, estimate flow at the coarsest level (where the motion is small in pixels), then warp and refine going down the pyramid.
Equivalent alternative the question hints at: down-sample first to shrink the pixel motion, solve, then up-sample and iterate — the same idea expressed as iterative refinement on a pyramid.

Solution — Naive Bayes classifier on pixel colour

When a single colour test (pure blue) fails, model the distribution of background colours and the distribution of foreground colours separately. Then for every new pixel, ask: "is this colour more likely to come from FG or BG?" That's Bayes.
  1. Extract features x from each pixel (e.g. RGB values, or a small colour histogram of the local patch).
  2. Background posterior: $p(B \mid x) \;\propto\; p(x \mid B)\, p(B)$, where p(x|B) is learned from the background pixels in the training images T and p(B) is the prior fraction of background pixels.
  3. Foreground posterior: $p(F \mid x) \;\propto\; p(x \mid F)\, p(F)$ — same construction on the foreground pixels.
  4. Classify pixel x as background if $p(B \mid x) > p(F \mid x)$, otherwise foreground.
Once a pixel is labelled background, replace it with the corresponding pixel of the computer-generated world. Foreground pixels (the actor) are kept. This is exactly the recipe the answer key gives.

Final · Question 4 — Fast R-CNN → Faster R-CNN & Mean Shift (17 marks)

i. Function of the ROI pooling layer

The fully-connected layers after the backbone need a fixed-size input. But region proposals come at all shapes and sizes. ROI pooling is the gadget that takes any rectangular crop of the feature map and outputs a tensor with a fixed spatial size (e.g. 7×7).
ROI pooling resizes / warps every region proposal to a fixed feature-map size by dividing the proposal into a fixed grid of cells and max-pooling inside each cell. This lets one shared classifier+regressor head work on regions of any original size.

ii. Sketch the Faster R-CNN modification

Faster R-CNN replaces the "selective search" ROI extractor (the slow, CPU-only part of Fast R-CNN) with a small Region Proposal Network (RPN) that shares the backbone features. Conceptually:
  • Backbone CNN (shared) → feature map.
  • RPN slides over the feature map, predicting objectness + box offsets for K anchors per location.
  • Top-N RPN proposals → ROI pooling on the same feature map → FCs → softmax classifier + bounding-box regressor.

iii. Main changes between Fast R-CNN and Faster R-CNN

  1. ROI extractor is learned. Selective search (hand-engineered, slow, runs on CPU) is replaced by the RPN — a small CNN head that produces proposals from the backbone feature map.
  2. Single network, end-to-end trainable. RPN and detector share the same backbone and are trained together with a multi-task loss.
  3. Speed. Faster R-CNN runs near real-time because proposal generation is no longer the bottleneck.
  4. Anchors. Proposals are generated from a small set of anchor boxes (different scales × ratios) at every feature-map location — a fixed, dense, learnable grid instead of a few thousand selective-search rectangles.

i. Mean Shift for image segmentation — main steps

Mean Shift is a mode-seeking algorithm: every pixel "climbs" toward the densest spot in feature space (colour, or colour+position). Pixels that climb to the same peak are assigned to the same segment.
  1. Find features for every pixel (colour, gradients, texture, …). A common choice is (L, a, b) colour + (x, y) location.
  2. Initialise windows centred at each feature point (each pixel).
  3. Perform mean shift for each window: shift the window's centre to the mean of the points inside it, then repeat until the centre stops moving (converges to a local mode).
  4. Merge windows that converged to the same peak — those pixels form one segment.

Parameter that must be set:

Window (bandwidth) size — the radius of the search window in feature space.

ii. Sensitivity to the bandwidth

Think of the bandwidth as the "smoothing strength" applied to the colour density. Too big → everything merges. Too small → every speck is its own segment.
BandwidthEffect
Very largeFast but not accurate. Distant points are pulled into the same mode, so distinct objects get merged into one segment (under-segmentation).
Very smallSlower but more accurate — though if it is too small, noise pixels become their own modes and you get over-segmentation: noise detected as separate "objects".
Practical choice: pick a bandwidth large enough that pixel-level noise gets averaged out, but small enough that distinct objects stay in different modes.

Final · Question 5 — RANSAC line fit & CBIR Average Precision (13 marks)

i. RANSAC for the line-fitting task

RANSAC's recipe in one sentence: guess a model from a tiny random sample, score it by counting points that agree, repeat many times, keep the best. For line fitting the "tiny sample" is 2 points — that's the minimum to define a line.
  1. Sample 2 points uniformly at random from the point cloud.
  2. Solve for the line passing through those 2 points (model parameters).
  3. Score by counting how many of the remaining points lie within a preset distance threshold of this line — those are the inliers.
  4. Repeat steps 1–3 for N iterations; keep the line with the highest inlier count.
  5. (Optional refinement.) Re-fit the line with least-squares on all the inliers of the best model.

📚 L06 — RANSAC (lec 6, slide 24 per answer key).

ii. Minimum number of iterations for 60% inliers, P = 0.9

The standard RANSAC formula tells you how many random samples you need before you can be at least P confident that at least one sample contained no outlier: $$N \;>\; \frac{\log(1-P)}{\log\!\big(1 - (1-e)^{s}\big)}$$ where e = outlier fraction, s = points per sample (= 2 for a line), P = required confidence.
Plug in the numbers:
e = 1 − inliers = 1 − 0.6 = 0.4
P = 0.9
s = 2 (a line needs 2 points)
$$N \;>\; \frac{\log(1-0.9)}{\log\!\big(1-(1-0.4)^{2}\big)} \;=\; \frac{\log(0.1)}{\log(1-0.36)} \;=\; \frac{\log(0.1)}{\log(0.64)} \;\approx\; 5.15$$
Round up — you need at least N = 6 iterations to hit the 90% confidence target.

i. The "semantic gap" in CBIR

CBIR retrieves images using low-level visual features (colour histograms, textures, deep embeddings). Users, however, query with high-level intent ("show me playful cats"). The mismatch between those two worlds is the semantic gap.
Semantic gap: the difference between what the user means / intends and what can actually be extracted from the image using the available features. (Lec 8, slide 35.)

ii. Average Precision (AP) for both models — K = 6, GTP = 3

For a ranked list of K results, AP@K = (1/GTP) × Σ (Precision@i) over only the positions i where a relevant ("cat") result was retrieved. Non-relevant positions contribute 0 to the sum.
Formula: $\displaystyle AP_K \;=\; \frac{1}{GTP}\sum_{i=1}^{K}\frac{TP_{seen}}{i}$ where the sum runs over all K positions but only relevant ones contribute. K = 6 (results count), GTP = 3 (only 3 cat images exist in the database).
