Midterm · Question 1 — Identify what each 3×3 mask does

Solution

A mask with corners = 0, four sides = −1, and the centre = a positive number is a Laplacian-family kernel. The size of the centre tells you which one:
  • centre = 4 → the Laplacian itself → edge detector.
  • centre = 5 (= 4 + 1) → Laplacian + identity → sharpening mask.
  • centre = 8 → 8-neighbour Laplacian (the diagonal version).
  • centre = 9 → 8-neighbour Laplacian + identity → sharpening (diagonal).
Matrix 1 (centre = 4) → detects edges. It produces a near-zero response in flat regions and a large response wherever the brightness changes.

Matrix 2 (centre = 5) → sharpening, keeps shapes. Think of it as "original image + edges": the +1 added to the centre says "keep everything you had", the rest of the kernel adds the edge map on top.

Midterm · Question 3 — Negative & γ on pixel values 10, 20, 100, 200

Solution

Three quick rules:
  • Negative: subtract from 255. Dark becomes light, light becomes dark.
  • γ < 1 (e.g. 0.2): brightens the image. Dark pixels lift the most.
  • γ > 1 (e.g. 2): darkens the image. Bright pixels drop the most.

Steps to fill the gamma row: divide the pixel by 255 → raise to γ → multiply by 255 again.

r = 10r = 20r = 100r = 200
Negative (255 − r)24523515555
γ = 0.2≈ 133≈ 153≈ 211≈ 243
γ = 2≈ 0.4≈ 1.6≈ 39.2≈ 156.9

Sanity check on the trends: in the γ = 0.2 row every value jumped up (10 → 133, 200 → 243). In the γ = 2 row every value fell (200 → 157, 10 → 0.4). That's the pattern you should expect every time.

Comment: increasing γ ⇒ image gets darker. Decreasing γ ⇒ image gets brighter. (The answer key's phrasing: "γ increase ⇒ brightness decrease, and vice versa.")

Midterm · Question 4 — Frequency-domain mini-questions

Part 1 — Difference between low-pass and high-pass filter

An LPF blurs (you can still see the shape but not the edges). An HPF outlines (you only see the edges, the interior goes flat/dark). The silhouettes in the picture show exactly that — solid blobs are LPF, the outline is HPF.
Low-pass (LPF)High-pass (HPF)
Lets throughLow frequencies (smooth regions)High frequencies (edges, fine detail)
DropsHigh frequenciesLow frequencies
Effect on imageBlur / smooth / remove noiseSharpen / show edges

Part 2 — Three images: how do you make the third from the first two?

If image 3 is just the outline of image 1, and image 2 is a slightly shrunken (eroded) version of image 1, then image 3 = image 1 − image 2. That's the morphology operation called boundary extraction.
Answer: Boundary extraction — the original minus its erosion gives the one-pixel-thick contour.

Part 3 — Multiplying the spatial image by (−1)x+y

Multiplying by (−1)x+y = flipping the sign on a checkerboard pattern of pixels. The effect on the Fourier spectrum is that the DC term moves from the corner to the centre — exactly the "fftshift" trick used to display spectra nicely.
Answer: Shifts the spectrum so the DC (zero-frequency) term lands at the centre.

Midterm · Question 5 — SIFT

1. Why do we use rotation invariance?

So a feature is matched correctly even when the same object is rotated. Without it, every rotated copy would need its own template — useless for object matching in the real world.

SIFT solves it by giving every keypoint a "canonical direction" (the dominant gradient orientation in its window), then describing the patch relative to that direction.

2. Why apply smoothing before filtering?

Derivatives amplify whatever is high-frequency. Noise is high-frequency. So if you don't blur first, your edge detector will fire on the noise instead of the real edges.
Answer: to remove noise first (Gaussian blur) so the derivative step picks up real edges, not noise.

3. The scale-space / DoG figure

The figure is the cascade Gaussian → Gaussian → Gaussian → … with increasing blur σ. The differences between adjacent blurred images (DoG) approximate the Laplacian-of-Gaussian (NLoG) at each scale — and that's what SIFT searches for blob-like keypoints in.
Key idea: DoG ≈ scale-normalised Laplacian-of-Gaussian. You take it because it's much cheaper than computing the real LoG, and it gives you keypoints that are stable across scale.

Final · Question 1 — Compare FCN and Mask R-CNN

Solution

FCN gives you one big colouring book page — every pixel gets a class. Mask R-CNN gives you a separate sticker per detected object. So if you photograph two cars: FCN paints both "car"; Mask R-CNN gives you two distinct masks.
FCNMask R-CNN
Type Semantic segmentation — labels every pixel with a class, but doesn't separate two objects of the same class. Instance segmentation — produces a separate binary mask per detected object.
Output structure One full-image map sized H×W with a class score per pixel. Per detected object: class label + refined box + a small mask (e.g. 28×28) for that one instance.
Output-layer activation Softmax over classes per pixel — every pixel commits to one class. Sigmoid per pixel of each class channel — mask and class are independent (classification head separately picks the channel).

Final · Question 2 — YOLO loss (v2 or higher)

The loss has three named pieces

You can always remember the three pieces by what they answer: where is the box? + is there really an object here? + which class is it?
NameWhat it measures
1. Localisation loss How wrong the predicted box coordinates (centre x, centre y, width, height) are compared to the ground-truth box. Squared / smooth-L1 error.
2. Objectness (confidence) loss How wrong the model is about whether there is an object in this cell/anchor. Binary cross-entropy; the target is the IoU with the GT.
3. Classification loss How wrong the predicted class probabilities are. In v3+ it's an independent binary cross-entropy per class (so multi-label is possible).
Total YOLO loss = localisation + objectness + classification, summed only over the "responsible" cells/anchors (the ones that actually contain an object).

Final · Question 3 — GIoU vs IoU vs DIoU

1. Advantage of GIoU over IoU

IoU is zero whenever the two boxes don't overlap, no matter how far apart they are. That means the gradient is also zero → the optimiser has no idea which way to push the box. GIoU fixes this by also looking at the smallest box that contains both.
GIoU still gives a non-zero loss / gradient when the boxes don't overlap, so training can pull a far-away predicted box toward the ground truth. IoU couldn't do that.

2. Advantage of DIoU over GIoU

GIoU still works "indirectly" — it shrinks the enclosing box. DIoU goes more direct: it pulls the centres of the two boxes closer to each other.
DIoU converges faster and lines the boxes up better, especially when the boxes are far from overlap or when their aspect ratios already match.

Final · Question 4 — R-CNN family & IoU loss

1. Why Fast R-CNN is faster than R-CNN (any two reasons)

R-CNN was slow for two stacked reasons: (a) it ran the CNN ≈ 2,000 times per image, and (b) it was actually three separate training stages saved to disk between steps. Fast R-CNN attacks both.
  1. Single CNN pass on the whole image. Fast R-CNN runs the backbone once and reuses the same feature map for every region proposal. R-CNN re-ran the entire CNN on each of ~2,000 proposals — most of that work was redundant.
  2. Single end-to-end network. Fast R-CNN combines classification and bounding-box regression into one network with one loss. R-CNN had three pipelines (CNN finetune → per-class SVMs → bbox regressors) with intermediate features cached on disk.

2. Why use 1 − IoU instead of IoU?

A loss should be small when the model is doing well. IoU is the opposite — bigger IoU means better. So we flip it: loss = 1 − IoU is 0 when boxes match perfectly and 1 when they don't overlap at all.
In short: training minimises the loss; the value that should be minimised needs to be small when you are doing well — that's why we subtract from 1.

Final · Question 6 — Output vector size on a 15×15 grid (80 classes, 10 anchors)

Solution

Think of one grid cell. For every anchor it has to predict 5 box values (object-present + 4 box coordinates) and 80 class scores. So one anchor = 5 + 80 = 85 numbers. Total per cell = anchors × 85.

1. Only 1 object per cell ⇒ 1 anchor active

Per-cell length = 1 × 85 = 85 numbers. Full tensor: 15 × 15 × 85 = 19,125 values.

2. Up to 10 objects per cell ⇒ 10 anchors active

Per-cell length = 10 × 85 = 850 numbers. Full tensor: 15 × 15 × 850 = 191,250 values.

Comment

More anchors per cell = more output channels, more parameters, and slower inference — but the model can now predict several objects of different classes in the same cell. That's exactly why YOLO v2+ kept "per-anchor classes" rather than sharing one class score across the whole cell.

Final · Question 7 — Choosing the correct number of anchor boxes

Solution — K-means on the GT box sizes

Don't guess anchors — let the dataset choose them. Run k-means on all the ground-truth widths and heights, sweeping K = 3, 4, 5, … and pick the smallest K that gives a "good enough" mean IoU. This is exactly the "Dimension Clusters" trick introduced in YOLO v2.
  1. Collect all GT (width, height) pairs from the training set.
  2. Run k-means for K = 3, 4, 5, 6, 7, 8, 9, … with 1 − IoU as the distance (not Euclidean — that would favour big boxes).
  3. For each K, assign every GT box to its nearest cluster and compute the mean IoU.
  4. Plot mean IoU vs K — it always rises and saturates.
  5. Pick the smallest K whose mean IoU exceeds your threshold (≥ 0.6 is the usual rule of thumb).
Result: the smallest K that's "good enough" — fewer anchors = faster and smaller network, but you don't want to drop below your acceptable IoU.

Final · Question 8 — Bilinear interpolation at (x, y) = (4, 7)

Solution

Bilinear interpolation = weighted average of the four corners. A corner that is closer to the point gets a bigger weight; a corner that is further away gets a smaller weight. The four weights always add up to 1.
Corners and their values: (3, 5) = 15, (9, 5) = 35, (3, 8) = 40, (9, 8) = 20.
Target point: (4, 7).
Step 1 — how far across in x? The point is at x = 4, between x₁ = 3 and x₂ = 9. So horizontally it sits 1 out of 6 of the way across → fraction a = 1/6. That means the left column gets weight 5/6 and the right column gets weight 1/6.
Step 2 — how far down in y? The point is at y = 7, between y₁ = 5 and y₂ = 8. So vertically it sits 2 out of 3 of the way down → fraction b = 2/3. The top row gets weight 1/3 and the bottom row gets weight 2/3.
Step 3 — multiply each corner by its row × column weight, then sum.

(3, 5) = 151/3 × 5/65/181575/18
(9, 5) = 351/3 × 1/61/183535/18
(3, 8) = 402/3 × 5/610/1840400/18
(9, 8) = 202/3 × 1/62/182040/18

f(4, 7) = 550 / 18 ≈ 30.56.
Sanity-check by intuition: the point is near the left column (a = 1/6 is small) and near the bottom row (b = 2/3 is big). So the bottom-left corner (40) should pull the value up the most — and it does, contributing 400/18 of the total 550/18.
